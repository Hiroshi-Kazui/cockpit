# claude-multi — マルチウィンドウ claude-cli マネージャ 要件・設計書

## 1. 目的

claude CLI を最大4つのペインで並行起動し、セッションごとのユーザ⇔エージェントのやり取りを完全な形で記録・アーカイブするデスクトップアプリケーション。収集したログは後日、開発を一任する疑似AGIハーネスの構築素材として利用する。

## 2. スコープ

本アプリの責務は「起動・表示・記録」に限定する。

- **含む**: ペイン管理、claude CLI のpty埋め込み、目的入力とタイトル生成、セッションログ（JSONL）のアーカイブ、トークン使用量の可視化
- **含まない**: 応答への満足・不満の評価UI（判断はアーカイブされたユーザ発言の内容から事後に行う）、会話内容への介入・加工、grill-me の起動制御（CLI側のスキル判定に完全に委ねる）

## 3. 技術スタック

- Electron + TypeScript
- Renderer: React + xterm.js（ターミナル描画）
- Main: node-pty（claude CLI のpty起動）、chokidar（JSONL監視）
- ストレージ: SQLite（better-sqlite3、セッションメタデータのインデックス）＋ ファイルシステム（JSONLアーカイブ本体）
- タイトル生成: `claude -p --model haiku` によるヘッドレス1ショット実行

## 4. 機能要件

### 4.1 ペイン管理

- 最大4ペイン。レイアウトは 1 / 2分割 / 4分割グリッドを切替可能
- 各ペインに「デフォルトフォルダ」を設定でき、設定はアプリ再起動後も保持される
- ペインごとに claude CLI を独立したptyプロセスとして起動（cwd = デフォルトフォルダ）
- CLI の対話UX（スラッシュコマンド、権限確認、スキル、フック）を一切損なわないこと。アプリは生のpty入出力を素通しする

### 4.2 起動フロー

1. ユーザがペインの「新規セッション」を押すと目的入力ダイアログを表示。**目的テキストの入力は任意**（空のまま開始できる）
2. 入力された目的テキスト（あれば）を保存し、対象フォルダで claude を起動
3. 目的テキストがある場合: 起動完了を検知後、目的テキストを最初のプロンプトとしてptyへ送信
4. 並行して `claude -p` で目的テキストから短いタイトル（claude.ai の「最近の項目」相当の粒度、20字程度）を生成し、ペインヘッダに表示
5. タイトル生成失敗時は目的テキストの先頭を切り出してフォールバック

**目的を空のまま開始した場合（初回発言からの目的決定）**: 初回プロンプトの自動送信は行わず、ユーザが自分でタイプする。アプリはセッションの JSONL を監視し、**最初のスラッシュコマンド／スキル呼び出し（`/` で始まる発言）ではないユーザ発言**を目的テキストとして採用する。採用後は通常どおり `claude -p` でタイトルを生成する。目的が決まるまでペインヘッダは「未設定」と表示する。ユーザが一度も発言せずセッションを終えた場合、目的は未設定のまま。

明確なオーダーは即プランニング、不明確な場合は grill-me を起動する分岐は、アプリではなくグローバルスキル grill-me の description と CLAUDE.md 側の指示で実現する（本アプリの関知外）。

### 4.3 テレメトリチャネル（statusLine フォワーダ）とセッション紐付け

各ペインの claude は `--settings` でアプリ生成の設定ファイルを指定して起動し、statusLine コマンドとしてアプリ付属のフォワーダスクリプトを登録する。Claude Code はUI描画（＝やり取りの発生）のたびにセッション情報のJSONをstdin経由でこのスクリプトへ渡すため、フォワーダはそれをローカルIPC（Unixソケット／名前付きパイプ）でアプリ本体に転送する。

このJSONには session_id、transcript_path（セッションJSONLの実パス）、model、コンテキストウィンドウ使用率、および Pro/Max サブスクライバー向けの rate_limits（five_hour / seven_day の used_percentage と resets_at）が含まれる（Claude Code v2.1.80 以降）。

- **セッション紐付け**: transcript_path をそのまま利用する。ディレクトリ監視による推定ロジックは不要
- **更新頻度**: やり取りのたびに自動送出されるため、ポーリングなしで「やり取りごとに最新」を満たす
- **環境の非汚染**: `--settings` によりペイン内のみ設定を上書きする。ユーザが既に独自のstatuslineスクリプトを設定している場合、フォワーダからチェーン呼び出しして端末内の表示を維持する

### 4.4 コンテキストのアーカイブ

- 紐付けたJSONLを追記監視し、アプリ管理のアーカイブディレクトリへ同期コピーする（元ファイルは一切変更しない）
- セッションごとにメタデータJSONを併置する: session_id、ペイン番号、目的テキスト（原文）、生成タイトル、cwd、開始・終了時刻、使用モデル、累計トークン
- SQLiteにセッション一覧をインデックスし、アプリ内から過去セッションを検索・閲覧できる
- アーカイブは追記のみ（append-only）。削除・編集機能は設けない。ハーネス素材としての完全性を優先する

### 4.5 使用量可視化（コンテキスト消費・残量）

- **ペインごと**: statusLine から届くコンテキストウィンドウ使用率を用い、ペイン内に**コンテキスト消費量**（compact が必要になるまでの目安）をゲージで表示する。使用率に応じて色が段階変化する: 〜59% 緑 / 60〜84% オレンジ / 85%〜 赤。やり取りのたびに即時更新。
  - なお JSONL の usage（input / output / cache read / cache write）集計＝累計トークンは、セッションメタデータとして §4.4／§5 に記録するが、**ペイン内の主表示はコンテキスト消費量ゲージ**とする（累計トークンの棒グラフは主表示ではない）。
- **ステータスバー（アプリ下部）**: statusLineフォワーダから届く rate_limits の実測値（Anthropicサーバ由来）を用い、次の2本を%＋棒グラフで表示。推定計算は不要
  - 5時間ウィンドウの残り = 100 − five_hour.used_percentage（resets_at からリセットまでの残り時間を併記）
  - 週次上限の残り = 100 − seven_day.used_percentage（同上）
- **更新タイミング**: いずれかのペインでやり取りが発生するたびに即時更新。全ペインがアイドルで一定時間（例: 5分）statuslineデータが届かない場合のみ、フォールバックとして非公開エンドポイント `GET https://api.anthropic.com/api/oauth/usage` を単発照会する。同エンドポイントは低頻度でも429を返す激しいレート制限が報告されているため、1分毎などの定期ポーリングは行わない
- **フォールバック表示**: rate_limits が空の場合（APIキーログイン時など）は、ローカルJSONL集計＋設定画面のプラン上限（プリセット: Pro / Max 5x / Max 20x、手動調整可）による推定表示に切替え、UI上に「推定」バッジを明記する

### 4.6 目的のライフサイクル

目的（purpose）はセッションよりも長寿命の第一級エンティティであり、**ユーザがアプリUI上で完了を明示するまで継続する**。

- 目的は空（未設定）のまま開始することもでき、その場合はセッションの初回発言から遡って決定される（§4.2）。決定までの間、目的は `active` だがテキスト・タイトルが未確定（ペインヘッダは「未設定」表示）の状態を取り得る
- ペインヘッダに「完了」操作を設ける。完了すると目的は `completed` となり、以後そのペインの新規セッションは再び目的入力ダイアログから始まる
- `/clear` や `/resume` によるセッション切替では目的は継続する。ペインで発生する各セッションは、そのペインの active な目的に紐付く
- **アプリ終了時**: active な目的は保持され、次回起動時に該当ペインへ復元される
- **再起動後の復帰（ワンクリック再開）**: active な目的を持つペインは目的タイトルと「再開」ボタンを表示する。押下すると同じ cwd で claude を `--continue` 付きで起動し、直前の会話を復元する。複数ペインの自動一斉起動は行わない（意図しないプロセス起動・トークン消費を避ける）

## 5. データモデル

```
purposes
  id            TEXT PRIMARY KEY   -- UUID
  pane          INTEGER
  text          TEXT               -- ユーザ入力の目的（原文）
  title         TEXT               -- 生成タイトル
  status        TEXT               -- 'active' | 'completed'
  created_at    INTEGER
  completed_at  INTEGER NULL

sessions
  id            TEXT PRIMARY KEY   -- claude の session_id
  pane          INTEGER
  purpose_id    TEXT NULL          -- purposes.id への参照
  origin        TEXT DEFAULT 'dialog'  -- 'dialog' | 'clear' | 'resume' | 'restart'
  purpose       TEXT               -- 目的原文（purposes.text の非正規化コピー）
  title         TEXT               -- 生成タイトル（同上コピー）
  cwd           TEXT
  started_at    INTEGER
  ended_at      INTEGER NULL
  jsonl_path    TEXT               -- アーカイブ先パス
  model         TEXT NULL
  tokens_in     INTEGER DEFAULT 0
  tokens_out    INTEGER DEFAULT 0
  tokens_cache_read   INTEGER DEFAULT 0
  tokens_cache_write  INTEGER DEFAULT 0

pane_settings
  pane          INTEGER PRIMARY KEY
  default_cwd   TEXT

app_settings
  key           TEXT PRIMARY KEY   -- plan_limit_5h, plan_limit_weekly など
  value         TEXT
```

## 6. マイルストーン

1. **M1**: Electronシェル、4ペイングリッド、node-pty で claude 起動、デフォルトフォルダ設定の永続化
2. **M2**: JSONL検出・紐付け・アーカイブ同期、セッションメタデータ保存
3. **M3**: トークン集計とペイン内棒グラフ、ステータスバーの5時間／週次残り表示（推定）
4. **M4**: 目的入力ダイアログ、初回プロンプト自動送信、`claude -p` によるタイトル生成、目的ライフサイクル（完了操作、再起動後の復元とワンクリック再開）
5. **M5**: 過去セッションの一覧・閲覧UI、レイアウト切替、磨き込み

## 7. リスク・制約

- JSONLの内部フォーマットは非公開仕様であり、claude CLI のアップデートで変わりうる。パーサは未知フィールドを無視する寛容な実装とし、バージョン追従を前提とする
- statusLineのJSONスキーマ（rate_limits 等）はバージョンにより変わりうるため、フィールド欠落を許容するパーサとし、欠落時は推定表示へ自動フォールバックする
- rate_limits はPro/Maxサブスクライバーのみ提供。APIキーログイン時は空になる
- フォールバック用の /api/oauth/usage は非公開仕様であり、予告なく変更・廃止されうる
- 4セッション並行はトークン消費が速い。ステータスバーの推定残量はあくまで作業計画の目安として扱う
